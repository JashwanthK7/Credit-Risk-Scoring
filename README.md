# Credit Risk Scoring

A production-grade machine learning system for evaluating loan default risk. Given a loan application, the system predicts the probability of default and returns an **APPROVED** or **REJECTED** underwriting decision based on a configurable compliance threshold.

The project covers the full ML lifecycle: dataset acquisition → exploratory data analysis → model training with hyperparameter tuning → experiment tracking → REST API serving → Docker deployment → React frontend.

## For Users — Running the API & UI

This section is for anyone who wants to **deploy and use** the credit risk scoring system without digging into the ML training code.

### Prerequisites

- [Docker](https://docs.docker.com/get-docker/) installed (for Option A), **or**
- Python 3.11+ installed (for Option B)
- A pre-trained model artifact (`artifacts/best_model.pkl`) — generated by running the training pipeline, or provided separately

---

### Option A: Run with Docker (Recommended)

This is the fastest way to get the API running. The Docker image contains only the serving dependencies (no training libraries), keeping it lean.

**1. Clone the repository**

```bash
git clone https://github.com/JashwanthK7/Credit-Risk-Scoring.git
cd Credit-Risk-Scoring
```

**2. Ensure the model artifact exists**

The model file must be present at `artifacts/best_model.pkl` before building the image. If you are receiving the artifact externally, place it there now. If you need to train it first, see the [Developer section](#for-developers--training-eda--experiment-tracking).

```bash
ls artifacts/best_model.pkl   # should exist before proceeding
```

**3. Build the Docker image**

```bash
docker build -t credit-risk-api .
```

**4. Run the container**

```bash
docker run -d \
  -p 8000:8000 \
  -e CREDIT_RISK_API_KEY=your-secret-key \
  --name credit-risk \
  credit-risk-api
```

- Replace `your-secret-key` with a secure key of your choice. If omitted, the default dev key `default-dev-key-123` is used (not suitable for production).
- The `-d` flag runs the container in detached (background) mode.

**5. Verify the API is running**

```bash
curl http://localhost:8000/
```

Expected response:
```json
{"status": "Online", "message": "Credit Risk API is actively running."}
```

**Optional: Load model from a remote URL at startup**

If you store your model artifact remotely (e.g. S3, GCS), you can pass its URL and the container will download it automatically on first start:

```bash
docker run -d \
  -p 8000:8000 \
  -e CREDIT_RISK_API_KEY=your-secret-key \
  -e MODEL_URL=https://your-storage-bucket.com/best_model.pkl \
  --name credit-risk \
  credit-risk-api
```

### Option B: Run Locally without Docker

**1. Clone the repository**

```bash
git clone https://github.com/JashwanthK7/Credit-Risk-Scoring.git
cd Credit-Risk-Scoring
```

**2. Create and activate a virtual environment**

```bash
python -m venv venv
source venv/bin/activate        # macOS/Linux
venv\Scripts\activate           # Windows
```

**3. Install serving dependencies**

```bash
pip install -r api_requirements.txt
```

**4. Set your API key (optional)**

```bash
export CREDIT_RISK_API_KEY=your-secret-key     # macOS/Linux
set CREDIT_RISK_API_KEY=your-secret-key        # Windows CMD
$env:CREDIT_RISK_API_KEY="your-secret-key"     # Windows PowerShell
```

**5. Start the API server**

```bash
python serve_api.py
```

Or with uvicorn directly (with auto-reload for development):

```bash
uvicorn serve_api:app --host 0.0.0.0 --port 8000 --reload
```

The API will be available at `http://localhost:8000`.

### Making Predictions

Send a POST request to `/predict` with the loan application data as JSON. You must include your API key in the `X-API-Key` header.

**Using curl:**

```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-secret-key" \
  -d '{
    "person_age": 28,
    "person_income": 55000,
    "person_emp_length": 3.0,
    "loan_amnt": 12000,
    "loan_int_rate": 11.5,
    "loan_percent_income": 0.22,
    "cb_person_cred_hist_length": 5,
    "person_home_ownership": "RENT",
    "loan_intent": "DEBTCONSOLIDATION",
    "loan_grade": "B",
    "cb_person_default_on_file": "N"
  }'
```

**Using Python (requests):**

```python
import requests

payload = {
    "person_age": 28,
    "person_income": 55000,
    "person_emp_length": 3.0,
    "loan_amnt": 12000,
    "loan_int_rate": 11.5,
    "loan_percent_income": 0.22,
    "cb_person_cred_hist_length": 5,
    "person_home_ownership": "RENT",
    "loan_intent": "DEBTCONSOLIDATION",
    "loan_grade": "B",
    "cb_person_default_on_file": "N"
}

response = requests.post(
    "http://localhost:8000/predict",
    json=payload,
    headers={"X-API-Key": "your-secret-key"}
)

print(response.json())
```

### API Reference

| Endpoint | Method | Description |
|---|---|---|
| `/` | GET | Health check — confirms the API is running |
| `/predict` | POST | Score a loan application and return default probability + decision |
| `/docs` | GET | Interactive Swagger UI (auto-generated by FastAPI) |

### Input Field Reference

| Field | Type | Required | Constraints | Description |
|---|---|---|---|---|
| `person_age` | integer | Yes | 18–99 | Applicant's age |
| `person_income` | integer | Yes | > 0 | Annual income in USD |
| `person_emp_length` | float | No | 0–59 (nullable) | Years of employment; omit or pass `null` if unknown |
| `loan_amnt` | integer | Yes | > 0 | Requested loan amount in USD |
| `loan_int_rate` | float | No | 0–29.99 (nullable) | Loan interest rate %; omit or pass `null` if not yet assigned |
| `loan_percent_income` | float | Yes | 0.0–1.0 | Loan amount as a fraction of annual income |
| `cb_person_cred_hist_length` | integer | Yes | ≥ 0 | Credit history length in years |
| `person_home_ownership` | string | Yes | `RENT`, `OWN`, `MORTGAGE`, `OTHER` | Home ownership status |
| `loan_intent` | string | Yes | `PERSONAL`, `EDUCATION`, `MEDICAL`, `VENTURE`, `HOMEIMPROVEMENT`, `DEBTCONSOLIDATION` | Purpose of the loan |
| `loan_grade` | string | Yes | `A`, `B`, `C`, `D`, `E`, `F`, `G` | Loan grade assigned by the lender |
| `cb_person_default_on_file` | string | Yes | `Y`, `N` | Whether the applicant has a historical default on file |

### Understanding the Response

```json
{
  "default_probability": 0.3142,
  "risk_status": "APPROVED",
  "compliance_threshold": 0.5
}
```

| Field | Description |
|---|---|
| `default_probability` | Model-predicted probability (0.0–1.0) that this applicant will default |
| `risk_status` | `APPROVED` if probability is below the threshold, `REJECTED` if at or above it |
| `compliance_threshold` | The cutoff value currently configured in `config/config.yaml` |

### What the Pipeline Does

`main.py` calls two high-level functions in order:

**`src/eda.py` → `run_eda()`**

Performs exploratory data analysis on the raw dataset and saves outputs (plots and reports) to the artifacts/EDA output directory. Details covered in the [EDA section](#eda-overview) below.

**`src/train.py` → `main_training_pipeline()`**

Orchestrates the complete ML workflow:
1. Loads and validates the raw data
2. Applies feature engineering and builds a preprocessing pipeline
3. Splits data into train/validation/test sets with stratification
4. Trains multiple candidate models with Optuna-driven hyperparameter search
5. Evaluates all models and selects the champion based on configured metrics
6. Logs all runs, parameters, and metrics to MLflow
7. Serializes the champion pipeline (preprocessor + model) to `artifacts/best_model.pkl`

---

### EDA Overview

The EDA module (`src/eda.py`) analyzes the raw dataset and produces the following:

**Target Distribution**
- Class balance analysis between default (`loan_status=1`) and non-default (`loan_status=0`) records
- The dataset is imbalanced (roughly 78% non-default, 22% default); this informs the use of class-weighted models and threshold tuning

**Numerical Feature Analysis**
- Distribution plots (histograms + KDE) for: `person_age`, `person_income`, `loan_amnt`, `loan_int_rate`, `loan_percent_income`, `cb_person_cred_hist_length`, `person_emp_length`
- Box plots comparing feature distributions across default vs. non-default
- Outlier identification (e.g. extreme ages or incomes)

**Categorical Feature Analysis**
- Count plots and default rate breakdowns for: `person_home_ownership`, `loan_intent`, `loan_grade`, `cb_person_default_on_file`
- Notably, higher loan grades (E, F, G) correlate strongly with higher default rates

**Correlation Analysis**
- Pearson correlation heatmap across numerical features
- Key finding: `loan_percent_income` and `loan_int_rate` show the strongest positive correlation with default

**Missing Value Report**
- `person_emp_length` and `loan_int_rate` contain missing values
- Missingness is handled in the preprocessing pipeline via imputation (see below)

All EDA plots and summary statistics are saved to the output directory defined in `config/config.yaml`.

---

### Feature Engineering & Preprocessing

The preprocessing pipeline is built using scikit-learn `Pipeline` and `ColumnTransformer` so it is fit only on training data and applied consistently to validation/test/production inputs — preventing data leakage.

**Numerical features** (`person_age`, `person_income`, `person_emp_length`, `loan_amnt`, `loan_int_rate`, `loan_percent_income`, `cb_person_cred_hist_length`):
- **Imputation**: Median imputation for `person_emp_length` and `loan_int_rate` (which have nulls)
- **Scaling**: Standard scaling (zero mean, unit variance)

**Categorical features** (`person_home_ownership`, `loan_intent`, `loan_grade`, `cb_person_default_on_file`):
- **Encoding**: Ordinal encoding for `loan_grade` (A=0 through G=6, preserving the natural risk ordering); one-hot encoding for the remaining categorical features

The full preprocessing + model object is serialized together as a single `Pipeline` in `artifacts/best_model.pkl`, so inference requires no separate preprocessing step.

### Models Trained & Hyperparameter Tuning

The training pipeline evaluates multiple candidate algorithms. Hyperparameters are optimized using **Optuna** with cross-validation on the training set.

**Candidate Models**

| Model | Notes |
|---|---|
| Logistic Regression | Baseline linear model; interpretable |
| Random Forest | Ensemble of decision trees; robust to outliers |
| XGBoost | Gradient boosting; typically strong on tabular data |
| LightGBM | Fast gradient boosting; handles large datasets efficiently |

**Hyperparameter Search (Optuna)**

Optuna runs a configurable number of trials (set in `config/config.yaml`) using the TPE (Tree-structured Parzen Estimator) sampler. Each trial proposes a different hyperparameter configuration, fits the model on training folds, and evaluates on a held-out fold.

Example search spaces:
- XGBoost/LightGBM: `n_estimators`, `max_depth`, `learning_rate`, `subsample`, `colsample_bytree`, `reg_alpha`, `reg_lambda`
- Random Forest: `n_estimators`, `max_depth`, `min_samples_split`, `min_samples_leaf`
- Logistic Regression: `C`, `solver`, `max_iter`

**Model Selection Metric**

The champion model is selected based on **ROC-AUC** on the validation set, which is appropriate given the class imbalance. Additional metrics logged per run include: accuracy, F1-score (weighted), precision, recall, and log loss.

### Experiment Tracking with MLflow

Every training run — including each Optuna trial — is logged to **MLflow**. The tracking database is the included `mlflow.db` SQLite file.

**Launch the MLflow UI**

```bash
mlflow ui --backend-store-uri sqlite:///mlflow.db --port 5000
```

Then open `http://localhost:5000` in your browser.

**What you can inspect in the UI:**

- **Experiments**: All training runs grouped by experiment name
- **Parameters**: Every hyperparameter tried per run (learning rate, depth, regularization, etc.)
- **Metrics**: ROC-AUC, F1, accuracy, precision, recall, log loss — per run and across runs
- **Artifacts**: Saved model files, confusion matrices, feature importance plots, SHAP summary plots
- **Run comparison**: Select multiple runs to compare metrics side-by-side in table or chart view
- **Best run**: Filter and sort by any metric to identify the champion configuration

**Viewing the champion model's parameters**

After training, the best model's run ID is printed to the console. You can also query it programmatically:

```python
import mlflow

mlflow.set_tracking_uri("sqlite:///mlflow.db")
client = mlflow.tracking.MlflowClient()

experiment = client.get_experiment_by_name("credit_risk_scoring")
runs = client.search_runs(
    experiment_ids=[experiment.experiment_id],
    order_by=["metrics.roc_auc DESC"],
    max_results=1
)

best_run = runs[0]
print("Best run ID:", best_run.info.run_id)
print("ROC-AUC:", best_run.data.metrics["roc_auc"])
print("Parameters:", best_run.data.params)
```

### Model Explainability with SHAP

After the champion model is selected, **SHAP** (SHapley Additive exPlanations) values are computed to explain model behavior globally and locally.

**Global feature importance (SHAP summary plot)**

Shows which features drive predictions most across the entire dataset. Typically the top contributors are:
- `loan_int_rate` — higher rates strongly predict default
- `loan_percent_income` — higher debt-to-income ratio increases risk
- `loan_grade` — lower-grade loans (D–G) are higher risk
- `cb_person_default_on_file` — prior default history is a strong signal
- `person_income` — higher income reduces default risk

The SHAP summary plot is saved to the artifacts/EDA output directory and logged to MLflow.

**Local explanations**

For any single prediction, SHAP waterfall plots can show exactly which features pushed the prediction toward or away from default. This is useful for regulatory explainability requirements (e.g. adverse action notices).

To generate a local explanation:

```python
import shap
import joblib
import pandas as pd

pipeline = joblib.load("artifacts/best_model.pkl")
preprocessor = pipeline[:-1]   # all steps except the final estimator
model = pipeline[-1]            # final estimator

sample = pd.DataFrame([{
    "person_age": 28, "person_income": 55000, "person_emp_length": 3.0,
    "loan_amnt": 12000, "loan_int_rate": 11.5, "loan_percent_income": 0.22,
    "cb_person_cred_hist_length": 5, "person_home_ownership": "RENT",
    "loan_intent": "DEBTCONSOLIDATION", "loan_grade": "B",
    "cb_person_default_on_file": "N"
}])

X_transformed = preprocessor.transform(sample)
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_transformed)
shap.waterfall_plot(shap.Explanation(values=shap_values[0], base_values=explainer.expected_value))
```

### Configuration Reference

All key settings live in `config/config.yaml`. Edit this file to change behavior without modifying source code.

```yaml
# Example structure — actual keys depend on your config/config.yaml
data:
  dataset_name: "..."          # Kaggle dataset identifier
  raw_data_path: "data/raw/"   # Where downloaded data is stored
  output_path: "artifacts/"    # Where trained models and EDA outputs are saved

training:
  test_size: 0.2               # Fraction of data held out for final evaluation
  val_size: 0.1                # Fraction of training data used for validation
  random_seed: 42              # For reproducibility
  cv_folds: 5                  # Cross-validation folds during Optuna search
  optuna_trials: 50            # Number of hyperparameter search trials per model
  optimization_metric: "roc_auc"

business_logic:
  compliance_threshold: 0.5    # Probability above which applications are REJECTED
```

**Tuning the compliance threshold:** Increasing `compliance_threshold` approves more applicants (higher recall, more risk); decreasing it is more conservative. This should be calibrated against your portfolio's risk tolerance and reviewed against the precision-recall curve logged in MLflow.
